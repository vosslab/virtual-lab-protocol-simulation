# Cell-culture-game cleanup: stabilize hood dispatch, then clean

## Title and objective

Stabilize the YAML walker (currently 3/25, failing at `add_trypsin`), then complete
the K-chain cleanup so the 83-file working tree can be staged into reviewable commits.

The handoff docs (`docs/HANDOFF_CURRENT_STATUS.md`, `docs/STATUS_2026-05-08.md`)
agree on the failure shape: the K2 schema migration is conceptually complete, but a
dispatch asymmetry in `src/scenes/hood.ts` makes `pbs_wash` pass and `add_trypsin`
fail despite identical step shape. No further refactor or vocab sweep should land
until that failure is pinned and fixed.

## Alignment with core philosophies

This plan follows the four repo-wide core philosophies (see
`docs/REPO_STYLE.md`):

- **Long-term over short-term.** The plan keeps the `completionPath` design
  and does not roll back to a top-level `interactionSequence` fallback just to
  make the walker greener. Patch 0 rebuilds and verifies before action so
  stale artifacts do not drive short-term fixes.
- **Fix the design, not the symptom.** The plan bans `step.id` walker
  branches, `window.set*ForWalker` back-doors, fake `source`/`destination`
  fields, direct `gameState` mutation, and synthesized interactions for
  `directTool` or `modal`. The hood dispatch fix must explain the
  `pbs_wash` -> `add_trypsin` asymmetry rather than masking it. If the bug is
  per-event-branch-specific, prefer extracting shared generic discharge
  handling over duplicating logic.
- **Fresh subagent per task.** Each work package names a fresh owner.
  Independent tasks get new agents with self-contained briefs. Reusing a
  large-context coder is avoided.
- **Atomic task decomposition.** Work splits into Patch 0 baseline, WP-T1
  trace, WP-H1 fix, WP-A1 audit, conditional beta/gamma/delta, cleanup,
  scrub, and staging. Each patch has one purpose and one clear verification
  step.

## Design philosophy

Pinned at the top to prevent drift. Each principle below names which core
philosophy it applies.

- **Stabilization-first** (Long-term over short-term). No cleanup, no rename, no dead-code deletion until the
  walker is fixed at the design level. "Back to the prior failure step" is a
  checkpoint, not
  success. Cleanup that lands on a broken walker hides regressions.
- **Fix the design, not the symptom** (Fix the design). No `step.id` branches, no `window.set*ForWalker`
  back-doors, no top-level `interactionSequence` fallback in the resolver, no
  synthesized fake interactions for `directTool`/`modal` steps. If the bug is
  per-completionEvent-branch-specific in `src/scenes/hood.ts:600-800`, prefer
  extracting shared generic discharge handling over duplicating logic between
  `pbs_wash`, `pipette_trypsin`, and `neutralize_trypsin`. Do not preserve a
  brittle event-switch when the trace shows identical shape handled differently.
- **Evidence before fix** (Fix the design). A fresh baseline rebuild precedes
  any trace. The hood
  asymmetry is investigated by trace (read-only instrumentation) before any
  production-code edit. No decision is made from stale `dist/` artifacts.
- **Atomic patches** (Atomic task decomposition; Fresh subagent per task).
  Each work package is one fresh agent, one outcome, one clear verification
  step (a verification target, not necessarily one shell command). The K-chain (K2a/b/c/d) is the precedent.
- **Humans commit.** AI agents stage, update `docs/CHANGELOG.md`, and stop. The
  maintainer reviews `git diff` and runs `git commit`.

## Stabilization plan (M0 only, per stabilization-first rule)

Walker failing at step 4 is a core failure. Per the skill, M0 is a stabilization
plan with at most 5 experiments and 2 success metrics. M1+ are blocked until M0
exits.

### Success metrics (step-id based; no hardcoded denominators)

The active protocol's step chain is derived from `PROTOCOL_STEPS` (emitted by
the build into `src/content/protocol_data.ts`). Numeric counts in the handoff
are historical observations only; they must not appear in durable acceptance
checks because split steps will change the denominator.

- **Metric A (M0 checkpoint):** the walker progresses past `add_trypsin` and
  reaches at least `count_cells`, which was the last known pre-resolver-compat
  blocker.
- **Metric B (M0 exit):** the `pbs_wash` -> `add_trypsin` dispatch asymmetry
  is explained in writing and fixed at the design level. Resolver and index
  tests are green. The walker either completes the active compiled protocol
  chain or fails later at a documented, unrelated blocker (e.g. a known K2
  modal gap such as `count_cells`).
- **Metric C (final acceptance, M1 or later):** the walker completes every
  step in the active compiled protocol chain. The denominator is derived from
  `PROTOCOL_STEPS` at run time, not hardcoded.
- **Metric D (no-regression):** resolver, index, coverage, graph-smoke, and
  game UI smoke tests stay green at whatever counts the current build emits.
  Counts captured in Patch 0 are the comparison baseline; do not assert
  hardcoded numbers in this plan.

### Experiments (bounded to 5; proof ladder)

1. **Side-by-side branch read.** Observed: walker fails at `add_trypsin`. Suspected:
   `pbs_wash` (line ~654) and `pipette_trypsin` (line ~669) per-event branches in
   `src/scenes/hood.ts` diverge. Single change: align the branches if divergence is
   trivial. Success: walker advances past `add_trypsin`. Revert: restore branch.

   Design guardrail: if the bug is event-branch-specific, prefer extracting
   shared generic discharge handling over duplicating logic across `pbs_wash`,
   `pipette_trypsin`, and `neutralize_trypsin`. Do not preserve a brittle
   per-event switch when identical-shape steps are being handled differently.

2. **Trace dispatch.** Observed: read does not pin the bug. Suspected: state
   transition between steps 3 and 4 misfires. Single change: add ONE
   `console.log({stepId, interactionIndexBefore, resolverKind, completionEvent,
indexAfter, triggerStepCalled, heldLiquid, selectedTool})` in
   `dispatchInteractionClick`. Success: log identifies the divergent line.
   Revert: remove log before any other change.
3. **State-reset hypothesis.** Observed: trace shows `interactionIndex` not reset.
   Suspected: defensive backstop in `game_state.ts:236-240` or `completeStep` reset
   on step entry misfires across step transitions. Single change: align reset
   semantics. Success: walker advances past `add_trypsin`. Revert: restore prior reset logic.
4. **Tool/heldLiquid hypothesis.** Observed: trace shows stale `selectedTool` or
   `heldLiquid`. Suspected: hood does not clear pipette state between identical
   steps. Single change: ensure clear-on-step-entry. Success: walker advances past `add_trypsin`.
5. **Stop and write a new plan.** Observed: experiments 1-4 fail; same
   architectural reason. Action: stop iterating; write a new dispatch-ownership
   plan in `docs/active_plans/` framed around the handoff's "Desired dispatch
   ownership model". Do not begin a refactor inside this plan. This plan
   closes; the new plan owns the redesign.

If experiments 1-4 succeed, stabilization closes after experiment 4 at most.
Experiment 5 is only triggered if 3+ prior experiments fail for the same reason.

### Stabilization checkpoint vs exit

- **M0 checkpoint (necessary, not sufficient):** walker progresses past
  `add_trypsin` and reaches at least `count_cells`. Step-id based; the
  denominator does not appear here. This proves the patch addressed the new
  asymmetry; it does NOT close M0.
- **M0 exit (one of):**
  - **(a)** walker completes every step in the active compiled protocol chain
    (denominator derived from `PROTOCOL_STEPS`) with all other gates green;
    OR
  - **(b)** walker blocked at a later, unrelated step. The blocker must
    include: a named failing step id, a path to the `playthrough_report.json`
    capturing the failure, a one-paragraph cause summary, and a proposed
    owner / next task (created in `docs/active_plans/`). AND the
    `pbs_wash` -> `add_trypsin` dispatch asymmetry has a written root cause
    and a corresponding design-level fix landed (not a per-branch patch that
    masks the asymmetry). AND resolver, index, and coverage tests are green.

The active protocol's step count is read from `PROTOCOL_STEPS` in the build
output (currently emits the 25-step `cell_culture` revert state, but the count
shifts if alpha approves splits). Do not hardcode a numerator anywhere in this
plan or in tests; assert against named step ids and the chain length emitted
by the build.

- **In all cases:** all trace logging removed; `_temp*`/`_debug*`/`_inspect*`
  files deleted; `docs/CHANGELOG.md` records what was traced, the root cause,
  and the fix shape.

Reaching the prior failure step (`count_cells`) alone does not exit M0. Only
when Metric B is satisfied do M1-M3 begin.

## Migration and compatibility policy

- **Schema:** `completionPath` is the single source of truth. No top-level
  `step.interactionSequence` fallback. Validator Rule 8 stays strict.
- **Dispatch:** scene click handlers route `interactionSequence` through the
  resolver; `directTool` and `modal` are dispatched via scene click handlers
  and `data-walker-advance` attributes. The resolver does not synthesize
  interactions for non-`interactionSequence` kinds.
- **Legacy deletion criteria:** a branch in `src/scenes/hood.ts:955-1277` is
  deletion-eligible iff (a) every `interactionSequence` step now routes through
  the resolver path, proved by reading the routing in `onItemClick` and
  `dispatchInteractionClick`, and (b) no directTool or modal item id reaches
  the branch.
- **Backward compatibility:** none required; this is a single-author repo with
  no external consumers. Migration is one-way.

## Stop-and-ask conditions

Pause and ask only if one of these happens:

1. A new `completionPath.kind` is **required** -- not merely "potential".
   A small extension inside an existing kind (e.g. `kind: modal` gaining an
   `advancePlan` field that fits the existing modal model) is NOT a stop
   condition. Take the obvious extension, document the assumption, and
   continue.
2. The manager wants to roll back the `completionPath` design.
3. The proposed fix requires deleting non-generated user assets.
4. The trace points to a broad hood/bench dispatch refactor rather than a
   bounded fix.
5. The full gate suite cannot be made green after the documented patch
   sequence.
6. The plan would remove `devel/commit_changelog.py`. Keep it; the maintainer
   uses it manually.
7. The DOM cannot expose visible controls or input affordances for a flow
   without a user-facing UI change (e.g. requires keyboard-only input).
   Surface the exact missing affordance in this case.
