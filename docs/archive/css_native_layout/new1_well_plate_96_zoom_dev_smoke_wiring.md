# NEW1 well_plate_96_zoom dev_smoke wiring

## Purpose

This wiring connects the NEW1 CSS-native layout spike to a real, end-to-end runtime path
without touching curriculum content or production discovery defaults. It exists so that
a future agent or human can run the spike scene through the actual builder, runtime
bundle, and browser, and observe empirical evidence (feature-flag set count, viewport
presence, CSS-native invocation count) that the gate in
`adapter.ts`
fires on the renamed `well_plate_96_zoom_check_scene`. The rollback guarantee is that
every modified production surface is either a no-op-when-default (env-var off, flag off)
or a small contained namespace, and the removal recipe in this doc reverses every change
back to the pre-NEW1 state.

## Why dev_smoke only

`PRIMARY_SPEC.md` (section "Source-code and content layout")
declares `tests/content/dev_smoke/` as the explicit dev/test content root. Smoke
fixtures use the same YAML schema, builder, and walker path as curriculum mini-protocols
but are excluded from the student launcher and the full-protocol sequence. They declare
`protocol_type: dev_smoke` and are exempt from the learning-block requirement.

This wiring is NOT a production migration. It uses the new `INCLUDE_DEV_SMOKE` env-var
flag in `build_new_protocol_data.py`
to keep the default builder behavior unchanged. With the flag absent or unset, the
builder skips `tests/content/dev_smoke/` exactly as it did before NEW1 Task #67. The
spike scene is reachable only when an operator explicitly opts in by setting
`INCLUDE_DEV_SMOKE=true` for that one builder invocation.

## Build command

Run from the repo root in this order:

```
INCLUDE_DEV_SMOKE=true source source_me.sh && python3 pipeline/build_new_protocol_data.py
source source_me.sh && python3 pipeline/build_new_scene_data.py
source source_me.sh && python3 pipeline/build_object_data.py
bash pipeline/build_runtime_bundle.sh
```

The first command is the only one that needs the env-var flag. Subsequent
builders read the generated catalogs and do not re-scan content roots for
protocol discovery.

## Playwright command

```
cd <repo> && node tests/playwright/spike_built_app_well_plate_zoom.mjs
```

Expected: three assertions PASS:

1. feature-flag set count via `window.__spike.set_css_native_well_plate_zoom_spike_enabled_for_test(true)` returns and a follow-up `is_css_native_well_plate_zoom_spike_enabled()` reads `true` (set count >= 1);
2. the spike viewport DOM container is present after scene mount;
3. `window.__spike.get_css_native_invocation_count()` returns a value greater than zero, proving that the gate fired and `compute_scene_layout_css_native` executed.

## Contract risk audit

Each modified production file is audited against
`PRIMARY_CONTRACT.md` item 3 (clickable objects are
SVG-backed scene objects laid out by the layout engine; custom geometry only
for structured subparts).

- `INCLUDE_DEV_SMOKE` flag in `build_new_protocol_data.py`:
  NO contract risk. Default behavior preserved; the dev_smoke carve-out is
  explicitly sanctioned by `PRIMARY_SPEC.md`. The flag
  only widens the discovery set when explicitly requested by an operator.
- `__spike` namespace in `entry.ts`:
  NO direct contract risk. Contained, marked spike-only in its docblock,
  exposed only through a single named export with no behavior beyond
  re-binding existing feature-flag and counter accessors. It does not expand
  the authoring vocabulary or alter scene-object layout semantics.
- Scene rename to `well_plate_96_zoom_check_scene`:
  NO contract risk per se; the rename is cosmetic and consistent with the
  registered adapter id. Advisory note: if production code later begins
  treating the `_check_scene` suffix as semantic vocabulary (for example, to
  branch on dev_smoke vs curriculum at the layout layer), that would be a
  vocabulary drift and must be reviewed against
  `SCENE_VOCABULARY.md`. Today the
  suffix is only matched as a string literal in one gate.
- Counter in
  `feature_flags.ts`
  and increment in
  `css_native_adapter.ts`:
  NO contract risk while the flag default is `false`. The counter is
  unreachable from production hot paths in that state; only the spike test
  setter (via `__spike`) flips the flag on, and only the spike scene id
  passes the gate. The counter has no side effect beyond a module-local
  integer.

Conclusion: this wiring DOES introduce a small, contained, and removable
expansion of the production surface. Every introduced surface is either gated
by an env-var that defaults to off, gated by a hard-coded boolean that
defaults to false, or contained inside a single exported `__spike` namespace
that downstream production code never imports. The removal recipe above
reverses every change.

The pre-spike baseline reported in the task was 163 tsc output lines. The current count is 178,

### Spike-caused errors (category b)

All three errors were in `src/scenes/well_plate_96_zoom/well_plate_96_zoom.ts`:

- Line 28, col 45: `TS6133: 'itemId' is declared but its value is never read.`
- Line 28, col 61: `TS6133: 'ctx' is declared but its value is never read.`
- Line 40, col 32: `TS6133: 'ctx' is declared but its value is never read.`

Root cause: the no-op `dispatchWellPlate96ZoomInteraction` and `renderWellPlate96Zoom` functions
declare parameters to satisfy the interface shape but intentionally do nothing with them.

Fix applied: prefixed each unused parameter with underscore (`_itemId`, `_ctx`). This satisfies
`noUnusedParameters` without removing the parameters (which would break the interface) and
without using `as any` or `@ts-ignore`. After the fix, the spike-caused error count is 0.
