## K2 schema redesign - what it is and why it exists

Mid-M3 it became clear the original "every step has an interactionSequence"
model could not represent every step the protocol actually uses. Modal-driven
steps (drug additions, count cells, plate read) were being shoehorned into
fake one-element interactionSequences. SP-K1 redesigned this.

The result is `step.completionPath`, a discriminated union with three kinds:

- `kind: "interactionSequence"` - ordered click chain. Has `interactions: [...]`.
  Used for hood liquid transfers (spray_hood, aspirate, pbs_wash, add_trypsin,
  neutralize_trypsin, centrifuge feeders, resuspend, seed_plate).
- `kind: "directTool"` - single-click action. Has `tool` and `completionEvent`.
  Used for spray bottle, centrifuge, incubator placement, water bath, MTT
  decant, etc.
- `kind: "modal"` - opens a UI modal that emits an advance event. Has
  `openClick`, `advanceClick`, `completionEvent`. Used for cell counting,
  drug dilution prep and addition, plate reader, results screen.

Subpatch chain that landed K2:

- SP-K1 - vocab and schema docs.
- SP-K1b - locked the three-kinds decision after a brief flirtation with a
  fourth `instrument` kind. Modal advance buttons identified by a kebab-case
  `data-walker-advance` attribute, not by an items.yaml id.
- SP-K2a - additive type and parser support; legacy fields still accepted.
- SP-K2b - YAML migration of all 25 cell_culture steps plus the two tutorials.
  Top-level `interactionSequence` and `completionTrigger` removed; replaced
  with `completionPath`. Builder derives `completionTrigger` for runtime use.
- SP-K2c - Validator Rule 8: `completionPath` is required, kind discriminator
  is required, per-kind required and banned fields are enforced, legacy
  fields are hard errors. K2a's transitional compat read removed.
- SP-K2d - walker dispatch rewired to switch on `completionPath.kind`.
  Generic handlers `walkInteractionSequence`, `walkDirectTool`, `walkModal`
  replace per-step recipes. `data-walker-advance` attributes added to modal
  advance buttons. Walker drives all 25 steps schema-only.

## SP-K2bd2: the failed attempt and the revert

After K2d, an attempt was made (SP-K2bd2) to handle a real authoring need:
some "modal" steps had a setup interaction (e.g., select dilution choice) and
then a confirm interaction (e.g., click the "add to wells" button) that
together should be one step but were trying to be one modal kind. The
proposed solution was to split such steps into `_setup` + `_confirm` pairs.

That attempt went off the rails:

- The coder split 7 steps in YAML into 14 (count_cells plus 6 drug pairs).
- Walker support required a `setQuadrantCountsForWalker()` back-door
  installed on `window` to fake count-cells data without going through the UI.
- A `count_cells_confirm` walker special case manually invoked that back-door.
- 12 split-step IDs were registered in `src/steps/drug_treatment.ts` with
  `_setup` and `_confirm` event emitters.
- The walker stayed at 9/25 steps green - only count_cells passed, the rest
  failed. The "fix" was hiding behind step-id branches that contradicted the
  K2 design philosophy ("fix the design, not the symptom").

Decision: revert SP-K2bd2 entirely. The revert (task #76) is now complete:

- Step count back to 25.
- All `_setup` and `_confirm` step ids gone.
- `setQuadrantCountsForWalker` and its window export gone.
- Walker special case removed.
- All gates green except the YAML walker (which has a known count_cells
  ceiling carried over from K2d).

The work was then to be redone as four atomic patches (alpha audit / beta
YAML splits / gamma walker rewrite / delta gates), tracked as #77 through #80.
Those tasks remain pending.

## SP-K2bd2-resolver-compat: the K2 integration gap

While diagnosing the SP-K2bd2 mess, the resolver test
`tests/playwright/test_interaction_resolver.mjs` was found failing 7/9 and the
sibling `test_interaction_index.mjs` failing 5/6. Both had been added during
the K2 chain but never run green.

Root cause:

```
generated data:  step.completionPath.interactions   (kind: "interactionSequence")
resolver reads:  step.interactionSequence           (top-level, does not exist)
```

The resolver had been wired against the pre-K2 schema. After K2a/K2b moved
the field, the resolver was returning `{kind: "no-op"}` for every step. This
predates SP-K2bd2 entirely.

The fix (task #81) - strict, no fallback:

1. Added `getInteractionSequence(step)` helper. Returns
   `step.completionPath.interactions` only when
   `step.completionPath.kind === "interactionSequence"`. Returns `null`
   otherwise (no synthesis, no top-level fallback, no leniency).
2. Both `resolveInteraction` (legacy callers) and `resolveInteractionByIndex`
   call the helper and return `{kind: "no-op"}` when it returns null.
3. The first revision attempt synthesized a fake one-element interaction
   array for `directTool` kind. That was rejected and removed in the
   revision pass - directTool dispatch happens via scene click handlers and
   `data-walker-advance` paths, not through the resolver. directTool is not
   the resolver's responsibility.
4. Two test fixtures (Test 5 in each resolver test) exercised the synthesis
   path and were updated: in `test_interaction_resolver.mjs` the spray_hood
   directTool test was deleted and replaced with one verifying directTool
   returns `no-op`; in `test_interaction_index.mjs` the spray_hood terminal
   test was deleted (count moved 6/6 to 5/5).

Final state of #81:

- Resolver and index tests fully pass.
- TypeScript clean, pytest clean, build clean, graph smoke 25/25, game_ui
  9/9.
- All hits of `interactionSequence` in `src/interaction_resolver.ts` are
  comments or the `kind === "interactionSequence"` discriminator. Zero bare
  property reads of `step.interactionSequence` remain.

## The blocker that surfaced after #81: dispatch asymmetry in hood.ts

Closing the K2 integration gap exposed a regression in the YAML walker:

- Pre-#81: walker reached step 8 (`count_cells`) before failing on the K2d
  count_cells modal gap.
- Post-#81: walker reaches step 3 (`pbs_wash`) and fails at step 4
  (`add_trypsin`).

The asymmetry is the puzzle. `pbs_wash` and `add_trypsin` have **identical
shape**:

- Same scene (hood).
- Same `kind: "interactionSequence"`.
- Same two-interaction click plan: pipette pickup -> load from bottle -> flask
  destination.
- Same tool (`serological_pipette`).
- Same destination (`flask`).
- Differ only in liquid (`pbs` vs `trypsin`), volume (4 vs 3 mL), and
  completion event (`pbs_wash` vs `pipette_trypsin`).

A fresh read-only Explore agent traced the dispatch in detail. Findings (with
citations):

- `src/scenes/hood.ts` does **resolver-first** dispatch. `onItemClick(itemId)`
  routes interactionSequence steps through `dispatchInteractionClick()` at
  line 791-792, which calls `resolveInteractionByIndex()` at line 548. All
  resolver result handlers (load, discharge, error, wrong_order) early-return
  at lines 556/599/612/781/785.
- `src/scenes/hood.ts:955-1277` contains legacy hardcoded item-id branches
  (e.g., `serological_pipette_with_trypsin + flask`) that are now **dead code
  for interactionSequence steps**. They were the only path pre-#81 because the
  resolver was returning no-op; post-#81 the resolver's early returns prevent
  them from ever firing for interactionSequence.
- `src/scenes/bench.ts` is structurally similar. Bench legacy branches handle
  centrifuge/water_bath/microscope/etc. (directTool/modal items), not
  interactionSequence - so no overlap there.
- `heldLiquid`, `interactionIndex`, and `triggerStep` are all mutated
  exclusively by the resolver pipeline (hood.ts:564-569 for load, 585 for
  index advance, 588/654/669/687/707/722/734/746/761 for triggerStep). The
  defensive backstop in `game_state.ts:236-240` refuses to complete a step
  if `interactionIndex < sequence.length`. `completeStep` resets
  `interactionIndex` to 0 on step entry (game_state.ts:251).

The Explore agent could not pin the exact failure cause without runtime
instrumentation. Three hypotheses, narrowing:

1. State reset misfires on the pbs_wash -> add_trypsin transition (the
   defensive backstop refuses to complete add_trypsin).
2. The completionEvent-switch in `hood.ts` around lines 600-800 has a per-
   event branch where `pbs_wash` (line 654) works but `pipette_trypsin`
   (line 669) does not. This is the most concrete hypothesis: the same shape
   passes through different branches identified by completionEvent, and one
   branch may have a bug.
3. The walker's per-step click loop has a tool-already-selected check that
   diverges between consecutive identical steps.

The structural finding stands regardless: pre-patch the walker reached step
8 only because the dormant resolver was being silently bypassed by the
legacy item-id branches. The new resolver path is correct and active; the
old fallback now dead-codes itself for interactionSequence; and any bug in
the new path is now visible instead of silently hidden.

The conclusion of the investigation pass: **keep #81, do not roll back, find
the bug in the new dispatch path before closing the patch.**

## Long-running engineering principles in force

These are saved as memory and applied across every dispatch in this session.
They should keep applying when work resumes:

- **Long-term over short-term.** Accept a small cost now to avoid larger
  costs later. Prefer the durable fix over the quick patch. (E.g.,
  redesigning the schema instead of adding fallbacks; rewiring the resolver
  instead of restoring a top-level alias.)
- **Fix the design, not the symptom.** When something behaves wrong, fix the
  design that allowed the problem. No try/except blocks to hide errors. No
  defensive defaults for required keys. No step.id branches in generic
  dispatch code. (The SP-K2bd2 revert was a direct application of this.)
- **Fresh subagent per task.** Each independent task gets a new Agent
  invocation with a self-contained brief. SendMessage to a heavily-loaded
  coder is fine for short clarifications, not for new work.
- **Atomic task decomposition.** Each subpatch has one owner, one clear
  outcome, one verification step. (E.g., the K2 chain split into K2a, K2b,
  K2c, K2d.)

## Glossary refresher

- **Step**: smallest unit of protocol progress; one entry under a part in
  `protocol.yaml`.
- **Completion path**: the step-level field that says how the step gets
  completed. One of three kinds.
- **Interaction sequence**: ordered list of interactions for an
  interactionSequence-kind step.
- **Interaction**: one logical player operation; 1-3 clicks.
- **Click plan**: ordered click list for one interaction. Tool first.
- **Completion event**: runtime signal that a step is complete. Emitted by
  the final interaction (interactionSequence) or by a directTool/modal
  handler.
- **Completion trigger**: step-level listener that fires on a completion
  event. Build-derived; authors do not write it.
- **Interaction index**: zero-based position in the active step's
  interaction sequence. Reset on step entry; advances as interactions
  complete.
- **Wrong-order click**: click that does not satisfy the current
  interaction. Soft-fail in gameplay; hard-fail in the walker.
- **Walker**: YAML-driven UI playthrough at
  `tests/playwright/e2e/protocol_walkthrough_yaml.mjs`. Real DOM clicks. The
  canonical regression test.
- **Graph smoke**: data-layer reachability test at
  `tests/playwright/protocol_graph_smoke.mjs`. Calls `completeStep` directly.
  Proves nothing about gameplay.
