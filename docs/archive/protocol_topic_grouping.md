# Plan: Protocol topic grouping under content/protocols/

## Context

`content/protocols/` currently holds every mini-protocol and sequence runner as a flat sibling folder. The listing is dominated by the `sdspage_*` cluster plus an unprefixed cell-culture cluster and a handful of sequence runners. The flat shape is hard to scan; new authors cannot tell at a glance which minis belong to which workflow.

The proposed layout groups by topic and pulls sequence runners out as siblings:

```
content/protocols/
  cell_culture/      mini-protocols for the OVCAR8 cell-culture / MTT workflow
  sdspage/           mini-protocols for the SDS-PAGE workflow
  runners/           sequence runners (cell_culture_full, routine_passage, sdspage_full)
```

This mirrors the established `content/objects/<kind>/` grouping pattern (see `docs/archive/objects_subfolder_grouping.md`). User concern is validation compatibility: the Explore audit confirmed 9 path-parsing touchpoints across `validation/`, `pipeline/`, and `tools/` that hard-code flat-layout assumptions. A naive `git mv` breaks focused validation, build-time discovery, and the walker tool. The plan handles the depth-sensitivity first so the actual folder move is mechanical.

User also requested an automated test that enforces the new hierarchy, so a future author cannot regress to flat layout or invent a fourth top-level cluster by editing YAML alone.

Predecessor pattern: `docs/archive/objects_subfolder_grouping.md` (precedent for grouping `content/objects/` by kind, archived).
Predecessor scratch: this scratch file previously held the YAML cleanup gate plan and the dream-YAML reauthoring plan; both have landed and been archived. The earlier draft is fully overwritten.

## Objectives

- Reorganize `content/protocols/` from flat to three-cluster topic grouping (`cell_culture/`, `sdspage/`, `runners/`).
- Make every protocol-discovery code path layout-agnostic (find protocols by the presence of a `protocol.yaml` marker, not by hard-coded path-depth indexing).
- Preserve every cross-protocol reference. Runners reference minis by `protocol_name` (semantic), not by path; no runner YAML edits required.
- Add an enforcement test (`tests/test_protocol_folder_layout.py`) that fails on any of: cluster set drift, depth drift, duplicate or missing `protocol.yaml`, `protocol_type`-vs-cluster mismatch, folder-name-vs-`protocol_name` drift, duplicate `protocol_name`, orphaned or shared sidecars, or discovery-vs-filesystem divergence.
- Document the new layout rule in the canonical spec surface (`docs/specs/TARGET_FILE_STRUCTURE.md`) and link to it from author-facing docs (`content/README.md`, `docs/FILE_STRUCTURE.md`).
- Keep `validation/validate.py`, `pipeline/bootstrap_generated.sh`, and `tools/run_protocol_walkthrough.py` green across the migration with no per-protocol special cases.

## Design philosophy

Two-step migration: code change first (make discovery layout-agnostic), folder move second. Trade-off: pays one extra patch up front in exchange for an atomic, reversible folder move that does not touch business logic. Alternative rejected (one-shot patch that moves folders and rewrites discovery in the same change): mixes mechanical and semantic edits in one diff, makes bisecting any regression harder, and forces every reviewer to validate the new discovery logic against the new layout simultaneously. Per `docs/REPO_STYLE.md` fix-the-design-not-the-symptom: marker-based detection (find ancestor with `protocol.yaml`) is robust to any future depth change, not just this one.

## Scope

- Rewrite path-parsing in every consumer enumerated by the Explore audit to be marker-based (find first ancestor directory containing `protocol.yaml`) instead of position-based (`parts[2]`).
- Convert every `*/protocol.yaml` and `*/scenes/*.yaml` glob to `**/protocol.yaml` and `**/scenes/*.yaml` (or equivalent `rglob` calls).
- `git mv` every protocol folder under `content/protocols/` into the three clusters.
- Add `tests/test_protocol_folder_layout.py` enforcing the closed cluster set, the relative-depth shape, the `protocol_type`-per-cluster rule, and the uniqueness of `protocol_name` across the tree.
- Add canonical layout rule to `docs/specs/TARGET_FILE_STRUCTURE.md`; cross-link from `content/README.md` and `docs/FILE_STRUCTURE.md`; refresh `docs/USAGE.md` only if it currently shows flat-layout examples.
- Update `docs/CHANGELOG.md` per patch.

## Non-goals

- Renaming any protocol (`protocol_name` fields untouched; cross-references remain valid).
- Changing the `protocol_type` enum (closed set unchanged).
- Editing runner YAML to point at new paths (runners reference by semantic `protocol_name`).
- Touching `generated/` outputs by hand (regenerated by `pipeline/bootstrap_generated.sh` after the move).
- Adding a fourth topic cluster. Three clusters cover current content; the enforcement test treats the cluster set as closed. A fourth cluster requires an enum edit to the test plus a spec edit to `docs/specs/TARGET_FILE_STRUCTURE.md`.
- Splitting or merging any mini-protocol.
- Touching `tests/content/dev_smoke/` (separate content root, unaffected).

## Cluster membership (closed set of cluster names)

The closed set is the three cluster names, not the protocol membership. Membership inside each cluster is open: new minis may be added freely as long as they land under the correct cluster.

| Cluster | Membership rule |
| --- | --- |
| `cell_culture/` | Every current mini-protocol in the OVCAR8 cell-culture / MTT workflow (passage, seeding, drug dilution, plate treatment, MTT prep / reaction / readout, trypan blue counting). |
| `sdspage/` | Every current mini-protocol with the `sdspage_` prefix. |
| `runners/` | Every protocol with `protocol_type: sequence_runner`. |

Cluster membership is reviewer-verified at M2 entry against the live tree, not against a hardcoded snapshot in this plan.

### Milestone M3: Enforcement + docs

- Depends on: M2.
- Workstreams: WS-ENFORCE, WS-SPEC, WS-DOCS.
- Entry criteria: M2 exit met.
- Exit criteria:
  - `tests/test_protocol_folder_layout.py` exists and enforces every rule below. Every rule cites `docs/specs/TARGET_FILE_STRUCTURE.md` in its assertion-failure message.
    1. **Closed cluster set.** Only three top-level subdirectories permitted under `content/protocols/`: `cell_culture/`, `sdspage/`, `runners/`. The set is a hardcoded enum in the test mirroring the enum in the spec.
    2. **Relative-depth shape.** Every `protocol.yaml` lives at exactly two path segments below `content/protocols/` (`<cluster>/<name>/protocol.yaml`). The check is computed relative to `content/protocols/`, not relative to the repo root. Exact-depth-one is intentional and is declared permanent in the spec; arbitrary deeper nesting is rejected even though marker-based discovery could resolve it. Rationale: a flat-inside-cluster shape keeps the listing scannable and matches the `content/objects/<kind>/<name>.yaml` precedent.
    3. **Exactly one `protocol.yaml` per leaf.** Each `<cluster>/<name>/` directory contains exactly one `protocol.yaml`. Duplicates (partial merges, accidental copies, stray backups) fail the test. No loose `protocol.yaml` sits directly under `content/protocols/<cluster>/` (it must be one segment deeper).
    4. **Type matches cluster.** `cell_culture/<name>/protocol.yaml` and `sdspage/<name>/protocol.yaml` have `protocol_type: mini_protocol`; `runners/<name>/protocol.yaml` has `protocol_type: sequence_runner`. No other `protocol_type` value is permitted under `content/protocols/` (e.g. `dev_smoke` belongs under `tests/content/`).
    5. **Folder name equals `protocol_name`.** The leaf folder name equals the `protocol_name` field inside its `protocol.yaml`. This prevents identity drift (folder `foo/` carrying `protocol_name: bar`) and makes the tree self-explanatory.
    6. **Unique `protocol_name`.** Every `protocol.yaml` carries a `protocol_name` that is unique across the whole `content/protocols/` tree. A duplicate fails the test with both offending paths in the message. This guards against an author copying a folder and forgetting to rename the inner `protocol_name`, which would otherwise pass discovery but silently shadow another protocol at runtime.
    7. **Sidecar ownership.** Every `materials.yaml` lives as a sibling of exactly one `protocol.yaml`. Every `scenes/` directory lives as a sibling of exactly one `protocol.yaml`, and every file inside a `scenes/` directory is a direct child (`scenes/<scene>.yaml`), not nested (`scenes/<sub>/<scene>.yaml`). No floating asset files, no asset files shared across two protocol folders, no `materials.yaml` or `scenes/` outside a valid protocol leaf.
    8. **Discovery round-trip.** The test imports the marker-based discovery helper from `validation/shared_toolkit/discovery.py` (or its successor), runs it against `content/protocols/`, and asserts that the set of discovered `protocol_name` values equals the set of folder names found by walking the filesystem directly. This closes the loop between filesystem layout, discovery code, and runtime identity: if discovery starts missing a protocol or inventing one, the test fails.
  - **Test structure.** Rules above are implemented as independent pytest functions inside `tests/test_protocol_folder_layout.py` so a failure points at one named concern, not a multi-page noise dump. Minimum function set: `test_cluster_set_closed`, `test_relative_depth_shape`, `test_exactly_one_protocol_yaml_per_leaf`, `test_type_matches_cluster`, `test_folder_name_equals_protocol_name`, `test_protocol_name_unique`, `test_sidecar_ownership`, `test_discovery_round_trip`. The shared per-protocol YAML-load is cached at module scope so the suite stays well under one second.
  - **Discovery round-trip uses the public entry point.** The round-trip check imports the same `discover_protocols()` function that `pipeline/build_protocol_data.py` and `tools/run_protocol_walkthrough.py` call, not an internal helper. This keeps the test resilient to refactors of the marker-walk implementation: any future change that keeps the public discovery contract passes; any change that silently breaks the contract fails the test before it breaks the build.
  - **Self-test fixture.** A tiny invalid fixture tree lives at `tests/fixtures/protocol_layout_negative/` and is exercised by `test_enforcement_test_rejects_bad_layout`, which asserts that running the layout rules against the fixture produces the expected failure for each rule. This prevents the enforcement test itself from silently degrading (a refactor that turns an assertion into a no-op is caught immediately).
  - WS-SPEC: `docs/specs/TARGET_FILE_STRUCTURE.md` carries the new "Protocol cluster layout" section with the full rule (closed set, depth, type-per-cluster, runner ownership). This is the only place the rule body appears.
  - WS-DOCS: `content/README.md` updated to show the three-cluster layout with a one-paragraph summary plus a link to the spec; `docs/FILE_STRUCTURE.md` updated; `docs/USAGE.md` updated only if it currently shows flat-layout examples.
  - `pytest tests/` exits 0.
  - `docs/CHANGELOG.md` close-out entry.
- Parallel-plan ready: yes -- WS-ENFORCE, WS-SPEC, and WS-DOCS are independent (test file, spec doc, author docs); max parallel doers 3.

## Migration and compatibility policy

- Additive at M1: marker-based discovery handles both layouts; no consumer breaks until M2.
- M2 is the breaking-layout change; it lands atomically (`git mv` operations grouped in one patch). Rollback is `git revert` of the M2 commits.
- No external consumers of these paths exist (curriculum is repo-internal).
- Enforcement test (M3) prevents regression: once landed, the layout is closed by the test.
- The spec rule in `docs/specs/TARGET_FILE_STRUCTURE.md` may be promoted to `docs/PRIMARY_CONTRACT.md` later if a persistent downstream consumer depends on the layout shape; until that trigger fires, the rule lives in `specs/`.

## Resolved decisions

- Enforcement test is split into named per-rule pytest functions inside one file. Reason: failure messages stay readable; each concern fails in isolation; future rule additions get their own named function. Single file keeps the discovery import and YAML cache shared.
- Discovery round-trip uses the public `discover_protocols()` entry point, not an internal marker-walk helper. Reason: decouples the test from implementation refactors; any future change preserving the public contract passes.
- Sidecar constraints (one `materials.yaml` per protocol; one `scenes/` directory per protocol; no nested scene sub-directories) are declared as design rules in `docs/specs/TARGET_FILE_STRUCTURE.md`, not just as test artifacts. Reason: future authors see the constraint as intentional, and a refactor that wants to relax it must edit both the spec and the test.
- M2 includes a pre-move WS-PRECHECK that verifies folder name equals `protocol_name` before WS-MOVE runs. Any misalignment is fixed in a small dedicated rename patch (Patch 2a) so WS-MOVE itself stays purely mechanical.
- A negative fixture under `tests/fixtures/protocol_layout_negative/` exercises every rule's failure path inside CI. Reason: prevents the enforcement test from silently degrading; replaces the earlier manual planted-failure step.
- Two-step migration (code before move) over one-shot. Reason: bisectable, atomic move.
- Marker-based discovery (`find ancestor with protocol.yaml`) over position-based (`parts[N]`). Reason: robust to future depth changes; fixes the same family of bug as the recently-patched `discovery.py` off-by-one.
- Three-cluster layout (`cell_culture/`, `sdspage/`, `runners/`) over five (per-pathway) or two (mini/runner). Reason: aligns with the only two pedagogical workflows currently authored, with runners pulled out so the topic clusters contain only mini-protocols.
- Sequence runners moved into `runners/`, not kept alongside their topic cluster. Reason: a runner that spans clusters (e.g. a future cross-topic runner) has no natural home in any one cluster; the dedicated `runners/` folder side-steps the ownership conflict.
- pytest enforcement (not validator entry) for the layout rule. Reason: the rule is structural (filesystem layout, not YAML content); the existing `tests/test_no_file_folder_name_collision.py` is the precedent. Promotion to a `validation/structure/` stage is possible later if developers want to see the failure under `validation/validate.py`.
- Cluster set is closed in both the enforcement test and `docs/specs/TARGET_FILE_STRUCTURE.md`. Adding a fourth cluster requires editing both (forcing a deliberate review).
- No runner YAML edits. Cross-protocol references are by semantic `protocol_name`, verified by the Explore audit.
- `generated/` regeneration is a dedicated patch (WS-REGEN), not combined with the move patch (WS-MOVE). Reason: keeps the move patch purely mechanical and the regeneration patch purely artifact-update.
- Rule body lives only in `docs/specs/TARGET_FILE_STRUCTURE.md`; `content/README.md` and `docs/FILE_STRUCTURE.md` carry summary plus pointer. Reason: per `docs/PRIMARY_DESIGN.md` vocabulary-closure ("one canonical term per concept"), shadow rule statements drift and rot.
- Rule does NOT go into `docs/PRIMARY_CONTRACT.md` at this time. Reason: layout grouping is a structural convention, not a hard invariant. Same promotion-trigger pattern as the no-schema-version rule: promote only when a persistent downstream consumer depends on the shape.
