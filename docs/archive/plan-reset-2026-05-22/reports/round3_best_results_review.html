Executive summary

Round 3 began as a continuation of the no-crop fix queue inherited from Round 2 (approximately 101 cropped scientific SVG assets flagged by the static-template precheck). The first audit pass (A1) showed zero visible asset crops in runtime screenshots across the four mounted scenes. The whole premise of the queue was a measurement artifact — the static-template renderer hardcodes CSS classes that bypass the runtime sizing path. Round 3 pivoted from a no-crop campaign to a runtime-quality campaign on that single piece of evidence.

The largest single unblocker of the round was the foundation HTML wiring (#121). Before #121, walkers had no way to load a protocol the way a student would, so every cluster-scale measurement was either an internal-API call or a measurement bug masquerading as a runtime failure. Cluster mount audit, R9 rerun on 11, top-10 full walkthrough, and the OSC fire audit all became possible only after foundation landed.

The single most expensive lesson of Round 3 was R7: a runtime interaction smoke driver reported clickWorks=0/4 across the four mounted scenes. P5 then reported clickWorks=0/11 on an 11-protocol expansion. Both readings were driver divergence, not runtime regressions. The R9-corrected driver, rerun against all 11 protocols after foundation, recorded 11/11 mount + 11/11 clickWorks. The R7 paradox is now fully resolved.

The Foundation: HTML-Per-Protocol Wiring (#121)

Source: round3_foundation_html_wiring.md · PR: #121

The gap that blocked every cluster measurement

Before #121 the repo had a structural capability mismatch:

  • pipeline/build_protocol_html.py existed and could generate per-protocol HTML shells per the PRIMARY_SPEC convention, but it was never invoked from build_github_pages.sh.
  • src/scene_runtime/bundle/entry.ts exported loadAndMountByProtocolName() ready for IIFE bundle consumption, but no IIFE bundle was produced. Only the ESM dist/main.js existed.
  • Result: every Playwright walker that tried to load dist/<protocol>.html received 0/N mounted protocols. R9 and the cluster mount audit both confirmed this.

The shape of the fix

Two sequential build passes added after the existing ESM bundle in build_github_pages.sh. No source-code changes; the generator and the entry-point exports were both already correct. 

npx esbuild "src/scene_runtime/bundle/entry.ts" \
    --bundle --format=iife --global-name=SceneRuntime \
    --target=es2020 --platform=browser --minify --sourcemap \
    --outfile=dist/runtime.bundle.js

python3 pipeline/build_protocol_html.py --all

Each per-protocol HTML inlines its {protocol_name} in a <script type="application/json">, loads ./runtime.bundle.js, and calls SceneRuntime.loadAndMountByProtocolName(rootElement, protocolName).

dist/main.js2.3 MBESM bundle for landing page (unchanged)

Smoke verification

A smoke driver loaded dist/mtt_reagent_prep.html in headless Chromium and checked four conditions:

  • window.SceneRuntime is available (IIFE parsed).
  • window.SceneRuntime.loadAndMountByProtocolName is callable.
  • window.__RUNTIME_PROTOCOL_CONFIG is set after page settles.
  • The mount reports sceneId=mtt_reagent_prep_bench_workspace, entry step=pick_up_mtt_powder.

All four passed. Status: mount=yes.

The Pivot: From No-Crop to Runtime Quality

Sources: A1 (round3_runtime_truth_audit.md), R5 (decisions/static_templates_are_not_runtime_truth.md), C1 (round3_static_template_hardcoding_audit.md), C3 (round3_render_path_matrix.md)

The premise that died

Round 2 handed Round 3 a fix queue containing approximately 101 cropped scientific SVG assets. The crop counts came from the static-template precheck renderer. The Round 3 plan opened with Batch D as the continuation of that queue.

The audit that killed it (A1)

A1 built the production bundle, mounted each protocol via loadAndMountByProtocolName, waited 3.5 seconds for async mount, captured full-page screenshots at 1280x900, and counted visible cropped assets per the PRIMARY_DESIGN.md "Visual integrity" rule.

Result, across all four mounted scenes: zero visible asset crops on each. The actual runtime failure modes were oversized labels, small or placeholder assets, and label overflow past the viewport edge. None of them were crops.

The decision (R5)

R5 formalized what A1 had measured. Three rules:

  1. Static-template no-crop failures are not production runtime failures unless reproduced through runtime screenshots.
  2. Static templates can be used for visual experiments only.
  3. Runtime screenshots are authoritative for production quality.

The trust matrix (C3)

C3's render-path matrix rated the runtime as HIGH trust and the static template as MEDIUM trust. C1's audit explained why: static templates hardcode CSS classes that bypass renderPlacement in src/scene_runtime/render/scene.ts. The two renderers share SVG assets but not sizing logic, so they drift independently.

Baseline Reframing (A3)

Source: decisions/layout_manager_baseline_reframing.md

The framing error

Round 1 and Round 2 status reports framed the current procedural layout engine in src/scene_runtime/layout/layout_engine.ts as something close to a working visual baseline. Replacement candidates were judged "matches legacy" or "diverges from legacy." A3 formalized that this framing is wrong.

The required statements

  1. The current procedural layout engine is the current broken baseline, not a visual gold standard.
  2. The replacement is judged against visual and runtime gates, not "matches legacy."
  3. No-crop is the first forcing function.
  4. The project will not build "legacy forever" or "hybrid forever."

Why this matters for Round 3

The 101 cropped-asset count that drove the original Round 3 plan was produced by the same layout engine A3 declared broken. Legacy carries the failures, so legacy cannot be the reference.

Retirement gates

  • No-crop: zero visible crops on the corpus that currently shows ~101 crops under legacy.
  • Label readability: every authored label renders inside its placement card, at the authored size, with no overflow clip and no overlap.
  • Click-target stability: every interactive target named by a step's interaction is hit-testable through the visible SVG.

A3 and R5 compose: A3 says legacy is not the reference; R5 says static-template precheck is not runtime evidence. Together they force every Round 3 success claim onto a runtime screenshot.

The baseline-reframing note is the smaller of the two pivot decisions but it was the one that made R5 possible.

R7 Paradox Resolved: From 0/4 to 11/11

Sources: R7 (round3_runtime_interaction_smoke.md), R9 (round3_protocol_advance_repair.md), round3_p5_clickworks_diagnosis.md, round3_r9_rerun_on_11.md

The full arc

R7 reported clickWorks=0/4 across all four A1-mounted scenes. Five parallel rescue workstreams (P1, P2, P3, P4, R9) converged on the diagnosis: the runtime is healthy, the smoke driver was wrong. R9 fixed the driver and recorded 3/4 (the 4th case is gesture-mismatch, not a runtime failure).

The P5 expansion that re-opened the question

P5 then ran an "adapted from R7" driver against an 11-protocol expansion set. It reported clickWorks=0/11. The expansion implied the runtime might still be broken at scale, even after the R9 fix at small scale.

The diagnosis: DRIVER_DIVERGENCE

Audit round3_p5_clickworks_diagnosis.md ruled the P5 reading a measurement bug. P5's driver was "adapted from R7" but not from R9's corrected version. It carried all three bugs R9 had already fixed:

  1. Wrong state-field path. P5 read cfg?.world?.objects[k]?.state. R9 had documented that per-object state lives in world.objectStates. P5's statesChanged check could never fire.
  2. currentInteractionIndex absent from snapshot. P5 checked only stepChanged, sceneChanged, and statesChanged. The normal advance signal for interaction 0 of step 0 is currentInteractionIndex moving 0 to 1. P5 was blind to it.
  3. Missing force: true on clicks. Playwright's actionability heuristic rejects small SVG <g> targets even with non-zero bbox.

The post-foundation R9 rerun on 11

The R9-corrected driver was rerun on all 11 P5 protocols after foundation #121 landed. The driver was updated to load file://.../dist/<protocol>.html instead of an HTTP query parameter. Result:

  • Mount: 11/11. Every protocol loads its per-protocol HTML, parses the IIFE bundle, calls loadAndMountByProtocolName, and sets window.__RUNTIME_PROTOCOL_CONFIG.
  • clickWorks: 11/11. Every click on the active step's first target advances at least one of (stepDelta, interactionDelta, sceneChanged, statesChanged).
  • Console errors: 0. No mount errors, no click errors.

The lesson, sharpened

Two false-negative readings (R7 0/4, P5 0/11), one corrected driver, one foundation unblock, one 11/11 result. The runtime click-advance path has been healthy the whole round. The only thing that changed between 0/11 and 11/11 was the entry URL the driver targeted.

Lesson: a verification driver is software. When a driver reports total failure across a surface other drivers describe as working, suspect the driver and the entry URL first.

Well-Plate Viewport Fix

Source: round3_well_plate_viewport_bug.md

The bug

Two top-10 protocols failed at clicks on well-plate subparts: mtt_solubilization_readout blocked at step 0 interaction 3 (target well_plate_96.all_wells); plate_drug_treatment_media_adjustment blocked at interactions 3 and 5 (block_A_1_6 and block_B_H_1_6). Playwright reported "Element is outside of the viewport" even with force: true.

The diagnosis: SCENE_POSITION_OFFSCREEN

src/scene_runtime/render/scene.ts lines 67-82 set the SVG host to width="100%" height="100%" with viewBox="1 5 98 90" (aspect 98:90) and preserveAspectRatio="xMidYMid meet". At 1280px browser width the SVG resolves to 1175px tall, 31% taller than the 900px Playwright viewport. The well plate bottom maps to DOM y ≈ 849px plus chrome, exceeding the 900px hard bound. Force-click bypasses visibility checks but not coordinate bounds.

The fix

// src/scene_runtime/render/scene.ts L78-L80
- sceneRoot.setAttribute("width", "100%");
- sceneRoot.setAttribute("height", "100%");
+ sceneRoot.setAttribute("width", "1280");
+ sceneRoot.setAttribute("height", "900");

preserveAspectRatio="xMidYMid meet" then letterboxes the SVG within 1280x900. Post-fix plate bottom = (70.0 - 5) / 90 * 900 = 650px, well inside the viewport. Both previously-blocked protocols walked end to end on the rerun.

Deferred follow-up

The layout engine still defaults composite aspect to 1.0 (square) when no SVG file is present. Real SBS 96-well plates have aspect 0.667. Reading an aspect_ratio field from content/objects/plate/well_plate_96.yaml in buildAssetSpec would render the plate at the correct proportions. Filed for Round 4.

Two-line scene.ts edit; two protocols un-blocked; top-10 walkthrough advanced from 7/9 to 9/9.

// src/scene_runtime/render/scene.ts

Caps glyph height at 25% of placement width. One-way (never enlarges past the authored value). Per-placement.

#runtime-root .scene-viewport svg text,
#runtime-root svg text {
    paint-order: stroke fill;
    stroke: #ffffff;
    stroke-width: 0.35px;
    stroke-linejoin: round;
    font-weight: 600;
}

Why both shipped

R3 prevents overflow; R3-alt prevents low contrast against busy backgrounds. Complementary, not competing. The subjective improvement on sdspage_attach_lid_and_leads_workspace was rated 4/5.

The alias map that was rejected (A3-reject)

A draft workstream proposed introducing an asset alias map so that multiple authored names could resolve to a shared underlying SVG. It was rejected:

  • The current loader performs a literal "SVG_" + assetName.toUpperCase() lookup. No alias table exists in current source.
  • Older CHANGELOG entries reference an EQUIPMENT_ASSETS alias table and a getStaticSvg helper. Both were removed in a previous cleanup. Reintroducing alias machinery would re-create a vocabulary leak that was deliberately closed.
  • The semantic problem (multiple bottles, one SVG) is better solved by colormap-driven variants in the existing bottle asset.
Future contributors who find an older CHANGELOG entry referencing EQUIPMENT_ASSETS, getStaticSvg, or a src/svg_assets.ts facade should treat those as archaeological.

Round 3 Lessons Learned

Lesson 1: Foundation unblocks everything

Five cluster-scale measurements (mount audit 26/26, R9 rerun 11/11, top-10 walkthrough 9/9, OSC fire audit 5/5 mount, placeholder recount 119) became possible only after #121 wired per-protocol HTML into the build pipeline. The artifact itself is two passes in a build script; the leverage is the entire round.

Lesson 2: Drivers can lie at any scale

R7 said 0/4. P5 said 0/11. Both readings were driver-divergence artifacts. The R9-corrected driver, rerun post-foundation, recorded 11/11. A single corrected driver, on the right entry surface, undid both false negatives. When a single driver reports total failure across a surface that other drivers report as working, suspect the driver first.

Lesson 3: Premises can be falsified

Round 3 opened with a 101-crop fix queue. A1 measured runtime crops as 0. The entire premise was a measurement artifact. When a queue arrives from a previous round, audit its measurement basis before committing to its work items.

Lesson 4: Parallel competing experiments win

R3 (TS renderer clamp) and R3-alt (CSS halo) were initially framed as competing. They shipped complementary. A1B's per-object decisions beat the A3-reject alias-map proposal on the same problem. Running both in parallel and letting the evidence decide produced better outcomes than picking one upfront.

Lesson 5: Content gaps and code gaps look identical

R1 (code), R10 (content), and #129 (content typo) all addressed "scene won't mount" failures with the same outward symptom. The diagnosis path must check both branches.

Lesson 6: Vocabulary leaks cause recurrent drift

display_width_cm, EQUIPMENT_ASSETS, width_scale, and the alias map proposal all came from older docs and CHANGELOG entries that referenced source which no longer exists. V1 + V3 enumerate the leaks; R11 closed widthScale; V4 is the future sweep that closes the rest.
