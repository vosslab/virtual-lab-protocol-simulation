# Subpart addressing recommendation (architect memo)

Decision needed by: user (per cleanup-gate plan M1 close deadline).

Status: architect recommendation, awaiting user ratification. Spec
amendment to [../specs/OBJECT_YAML_FORMAT.md](../specs/OBJECT_YAML_FORMAT.md)
follows in a subsequent patch (Patch 2d) after ratification.

## Dream-YAML caveat

This subpart schema is a DREAM-YAML layout assuming ideal learning
conditions. It is target-state design, not a frozen eternal contract.
Revisable if real authoring pressure, runtime cost, or pedagogy
testing surfaces a problem. The structured-grid scope
(plates / racks / gels) is intentional, but the specific shape
(overlapping wells plus rows plus columns) is the current-best dream,
not the only possible dream. Treat it as a strong starting proposal
that the user, the protocol authors, and the stepper-runtime owners
may renegotiate when evidence supports a change.

## 1. Problem statement

Stepper triage ([yaml_cleanup_triage.md](yaml_cleanup_triage.md))
reports 478 raw `unknown_target_active_scene` warnings across 8 dirty
protocols, deduped to 219 unique signatures. Within that population,
`mtt_solubilization_readout.read_absorbance` alone emits **193 raw /
97 unique** signatures of shape `well_plate_96.<A1..H12>` (the same
step echoes a second time through `cell_culture_full`'s sequence-runner
walk; the runner's findings are not double-counted in the unique-sig
total but they ARE separately classified). Today every one of those
sigs is filed under S1 (authoring bug -- "well reference, missing
placement"). The classification is technically correct against the
current spec but it indicts the wrong layer: the author is doing what
the schema requires (address every well individually) even when the
real-world lab interaction is column-scale or row-scale.

User pedagogy framing (verbatim intent):

- a **single-channel pipette** acts at **well** granularity,
- an **8-channel multichannel** acts at **column** granularity,
- a **12-channel multichannel** acts at **row** granularity,
- a plate-wide reagent addition is a **plate-region** interaction.

Today's schema forces all four cases to be expressed as 96 individual
well interactions because `structure` only exposes one subpart
namespace (the wells generated by `name_pattern: "{row_letter}{col}"`).
Authors cannot say `well_plate_96.row_A` or `well_plate_96.col_3`;
they must enumerate `A1..A12` or `A3..H3`. The 193-line `read_absorbance`
step is the load-bearing example.

## 2. Proposed schema amendment (overlapping subpart namespaces)

Extend [../specs/OBJECT_YAML_FORMAT.md](../specs/OBJECT_YAML_FORMAT.md)'s
`structure` block so a single structured-grid object may declare
several subpart namespaces over one underlying SVG, with explicit
membership lists tying the higher-granularity namespaces back to the
canonical cells. Authoring surface stays closed: no free-form
metadata, no open string fields, no inferred groups.

Two shape options are presented. Architect recommends Option A.

### Option A (recommended): explicit subpart_groups block

The current `structure.name_pattern` and the generated wells remain
the canonical cell namespace. A new `structure.subpart_groups` mapping
declares additional addressable namespaces. Each group entry is keyed
by `group_kind` (closed enum: `row`, `column`, `region`) and carries a
closed list of members, each member naming a stable subpart id and an
explicit `contains` list of canonical cell names.

```yaml
object_name: well_plate_96
kind: plate

structure:
  subpart_kind: well
  layout: grid
  rows: 8
  cols: 12
  name_pattern: "{row_letter}{col}"
  subpart_groups:
    rows:
      group_kind: row
      members:
        - name: row_A
          contains: [A1, A2, A3, A4, A5, A6, A7, A8, A9, A10, A11, A12]
        - name: row_B
          contains: [B1, B2, B3, B4, B5, B6, B7, B8, B9, B10, B11, B12]
        # ... row_C .. row_H
    columns:
      group_kind: column
      members:
        - name: col_1
          contains: [A1, B1, C1, D1, E1, F1, G1, H1]
        - name: col_2
          contains: [A2, B2, C2, D2, E2, F2, G2, H2]
        # ... col_3 .. col_12
    plate_region:
      group_kind: region
      members:
        - name: all_wells
          contains: [A1, A2, ..., H12] # expanded fully in real YAML
```

Authoring rules baked into the schema:

- `subpart_groups` is OPTIONAL. Flat objects and structured objects
  that need no group addressing simply omit the block.
- The top-level key inside `subpart_groups` (here `rows`, `columns`,
  `plate_region`) is a free author-chosen label that must be
  snake_case; the load-bearing closure is the inner `group_kind` enum
  (`row | column | region`).
- Every name in `members[].contains` must resolve to a canonical cell
  generated by `name_pattern`; the build pipeline rejects unknown
  cells, duplicate group-member ids, and empty `contains` lists.
- A `region` group is the escape hatch for "the whole plate" and
  "a contiguous block of wells the protocol treats as one
  reagent-addition target". Region is bounded by the build-pipeline
  rule above (every contained cell must already exist); it is NOT
  an open authoring surface.
- The `kind` enum on `structure.subpart_kind` stays `well`; the new
  groups add row / column / region addressability without changing
  what each canonical cell IS.

### Option B (rejected): per-group native generators

A more compact variant would auto-generate the row / column lists
from `rows` and `cols` rather than asking authors to list members.
Rejected because (i) it implicitly extends the `name_pattern`
token set without an explicit author-visible declaration, (ii) it
introduces a second source of truth for membership (generator
plus protocol-side overrides), and (iii) it makes the "explicit
list" rule from the current schema's "Grouped targets are listed
explicitly" section harder to defend later when a non-rectangular
container (a 6-lane gel with a custom load order, a slot-rack
with reserved positions) reuses the same mechanism. Option A
keeps one rule -- "every group is an explicit member list" --
across plates, racks, and gels.

## 3. Protocol-side addressing and the cascade-write rule

Protocols address group members through the same dotted syntax used
for canonical cells:

- `target: well_plate_96.A1` -- a single well (unchanged).
- `target: well_plate_96.row_A` -- the row group.
- `target: well_plate_96.col_3` -- the column group.
- `target: well_plate_96.all_wells` -- the plate region.

Cascade-write rule for `ObjectStateChange` on a group target:

- The state change writes the group member's own subpart-state
  record AND propagates every declared field in the change to
  every cell named in the group's `contains` list.
- The propagation writes only fields the change explicitly names;
  it does not clear or default unmentioned fields.
- The stepper validates the cascade at write time: every named
  member must exist, every cell in `contains` must accept the
  field per the object's `state_fields`, and any per-cell write
  conflict (same cell named by overlapping groups in one step)
  is a build-time error.

This keeps the protocol vocabulary's "no named-group construct" rule
intact at the PROTOCOL layer: groups are declared on the OBJECT,
not invented in the protocol. The protocol still names one target.
The object schema is what fans the write out.

[PROTOCOL_VOCABULARY.md](../specs/PROTOCOL_VOCABULARY.md) likely
needs one targeted amendment to acknowledge that a `target` may
resolve to either a canonical cell or a declared group member.
[PROTOCOL_YAML_FORMAT.md](../specs/PROTOCOL_YAML_FORMAT.md) needs
the cascade-write rule spelled out in the `ObjectStateChange`
section.

## 4. Multichannel capability constraint

Add a narrow capability tag on pipette objects so the stepper can
warn when a single-channel pipette is asked to address a row or
column group, and when a multichannel is asked to address an
unsupported axis. Scope is tight: this is NOT a general capability
framework, just enough to express the four pedagogy cases.

Proposed shape: extend `state_fields` on a pipette object with a
new declared field, OR (preferred) add a closed-list capability
descriptor under `capabilities` that the stepper reads:

```yaml
# on a single-channel pipette
capabilities: [clickable, material_container, instrument_with_setpoint,
               cursor_attachable]
channel_addressing:
  channels: 1
  addressable_subpart_kinds: [well]

# on an 8-channel multichannel
capabilities: [clickable, material_container, instrument_with_setpoint,
               cursor_attachable]
channel_addressing:
  channels: 8
  addressable_subpart_kinds: [well, column]

# on a 12-channel multichannel
capabilities: [clickable, material_container, instrument_with_setpoint,
               cursor_attachable]
channel_addressing:
  channels: 12
  addressable_subpart_kinds: [well, row]
```

`channel_addressing` is one closed mapping with two closed fields
(`channels`, `addressable_subpart_kinds`). It is OPTIONAL; a pipette
object without it falls back to "well-only" (the safe default that
matches today's behavior). The stepper consumes this when a pipette
acts on a group target: it looks up `group_kind` on the target's
declared group, and emits ERROR if the pipette's
`addressable_subpart_kinds` does not include that group kind.

`region` group kind is intentionally NOT in any pipette's
`addressable_subpart_kinds`; a region write is for bulk reagent
addition by hand (bottle, reservoir), not for a channeled pipette.

## 5. Migration impact on the 193 warnings

Walk-through for the load-bearing case
`mtt_solubilization_readout.read_absorbance`:

Today: 96 interactions, each addressing one well
(`well_plate_96.A1` through `well_plate_96.H12`). Stepper raises 96
raw warnings per walk (97 unique sigs when the H-row sweep is
included; the additional sig is the plate-level ambiguity).

After amendment: the step is one `region` interaction, OR 8 row
interactions, OR 12 column interactions, author's choice per
pedagogy. The author of `read_absorbance` will likely choose 8
row reads (one per plate row, modelling the canonical multichannel
sweep) or 1 region read (the actual plate-reader behavior).

Expected post-amendment warning count for the well-reference
category:

- best case (single `all_wells` region per step) -- **0**
  remaining well-reference warnings on `read_absorbance`,
- typical case (8 row reads) -- **0** well-reference warnings;
  the 8 row targets are first-class declared members and resolve
  cleanly,
- worst plausible case (author retains some per-well interactions
  for genuine well-level pedagogy elsewhere) -- a handful, well
  under 10.

The same fix shape applies to `cell_culture_full`'s runner walk
through `read_absorbance` (same 96 cells); collapsing the leaf
step collapses the runner walk automatically. No separate runner
fix required.

Pessimistic estimate of total well-reference warning count
post-amendment, across all 8 dirty protocols: **<= 20 raw /
<= 10 unique** (down from ~193 raw / ~97 unique). Numbers
are estimates; actual count is measured after Patch 2d ships
the schema and the affected protocols re-author their plate
steps to use group targets.

## 7. Out-of-scope (explicit)

The following are NOT proposed, NOT implied, and NOT precedents for
later expansion without a separate ratified architect memo:

- A generalized hierarchical-subparts framework (nested groups of
  groups, recursive composition, parameterized membership
  generators).
- Arbitrary subpart compositions across multiple objects (no
  cross-object groups; a group is always inside one object).
- Subpart capabilities beyond the closed `group_kind` enum
  (`row | column | region`). No `quadrant`, `diagonal`,
  `checkerboard`, `outer_ring`, etc. in this pass. Each new
  `group_kind` requires its own architect memo and user
  ratification.
- Any addressability on flat objects (bottles, flasks, tubes).
- Any change to the layout engine
  ([../specs/LAYOUT_ENGINE.md](../specs/LAYOUT_ENGINE.md)); groups
  are an addressing concept, not a placement concept. The
  underlying SVG is unchanged.
- Any change to material modelling
  ([../specs/MATERIAL_CONVENTION.md](../specs/MATERIAL_CONVENTION.md));
  the cascade-write rule reuses the existing material fields per
  cell, it does not introduce group-level material state.
- Authoring affordances for racks and gels other than the same
  `subpart_groups` mechanism. Scope is "structured grids
  (plates / racks / gels)"; the mechanism is identical across
  those three object kinds when they need it.
