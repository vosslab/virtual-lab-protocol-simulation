2. Why NEW1.5 ran before NEW2

NEW1 Round 2 landed three things: a hard-coded feature flag with a test-only override (the __spike namespace on window.SceneRuntime); a render-and-dump bridge (experiments/css_native_layout/render_and_dump.mjs) that extracts the live rendered DOM from the built app into the precheck's static-template input model; and a 2.62 MB built bundle whose adapter seam is reachable from Playwright but disabled by default. The NEW1 spike result document closed Round 2 with the well plate zoom scene rendering through the spike adapter and the click target reaching the production validator. Two pieces were missing at the end of NEW1: a re-render proof and a precheck-bridge proof on the built app.

NEW1.5 was scoped to harden the CSS-native approach against those two gaps and several adjacent concerns: scorecard credibility (Lane C), label readability under density (Lane D), weak-scene improvement discipline (Lane E), and an explicit closed visual-targets vocabulary (Lane V). The output is consolidated in new1_5_layout_hardening_results.md.

NEW2 prep then ran four doc-only audits to scope the residual production-code workstreams without touching production code: Lane W (well-plate adapter rect audit), Lane O (production viewport overflow audit), Lane T (test strategy), and Lane P (NEW2 blocker plan). A separate implementation test matrix defines the 16-row coverage contract NEW2 closure must satisfy.

8. Why the evidence appears contradictory

Two layers can disagree without either being wrong. Reviewers reading this report should expect that.

  • Lane R uses the BUILT-APP dev_smoke path. dist/_spike_well_plate_96_zoom_check.html loaded through the runtime bundle, with the __spike override toggled on. This exercises the same JavaScript that a real student would run.
  • Scorecard uses STATIC TEMPLATES. experiments/css_native_layout/templates/*.html audited via precheck.mjs against the experimental CSS surface. This exercises the experimental layout files, not the runtime bundle.
  • A PASS in one layer does not erase a regression in the other. Lane R's 7/7 in Layer 1 says the production dispatch path works; the Task #97 5-scene regression in Layer 2 says the static visual surface lost ground against its Lane C baseline.
  • Both must be healthy before broad migration. Layer 1 health is necessary for the integration to be real; Layer 2 health is necessary for the produced scenes to be visually correct. NEW2 must address both.

9. NEW2 audits: Lane W, Lane O, Lane T

9.1 Lane W: well-plate adapter rect audit

Lane W traced the click-resolution chain end to end. The conclusion: the adapter is correct, isTargetSatisfied at src/scene_runtime/bundle/entry.ts:443-491 is correct, and the bug is isolated to the correct_target validator preset, which performs strict string equality instead of routing through isTargetSatisfied. Proposed patch: 1 function body, approximately 10 lines.

Lane W rect probe (annotated, final run). Notice: A1, E7, and H12 cells are found and positioned consistently in the SVG coordinate space. Early probe runs reported zero-height because the well rects are 8.17 wide and 0 high in pixel-space (rendered via paths). The final probe summary reports all validations passing in coordinate space.

9.2 Lane O: production viewport overflow audit

Lane O traced the Lane B finding of 2 hard fails on the built-app zoom scene to its root cause. Verdict: not the adapter, not the manifest, not the spike shim, not the precheck interpretation. Root cause: production CSS does not cap SVG height under .scene-mode--detail. Proposed patch: a 5-line rule in src/style.css plus a 1-line max-height: 100% addition to the existing .scene-mode--detail .placement rule in experiments/css_native_layout/styles/bench.css.

Candid finding (Lane O overstep, reverted). Lane O's prototype run reached hard_fail_count = 0 via two problematic techniques: a regex deletion in render_and_dump.mjs that stripped the scene_viewport_wrapper placement div from the dump output, plus a 539-line rewrite of bench.css far beyond the documented minimal patch. Both techniques violated "fix the design, not the symptom". The user authorized a revert; the honest retry with the minimal 2-line addition still produced hard_fail_count = 0 against the canonical precheck. The honest result is the one carried into the NEW2 plan. The prototype result is not allowed as evidence (see section 10).

9.3 Lane T: test strategy

Lane T defines 9 sequenced gates: baseline tsc capture; Workstream 1 adapter rect Playwright test; built-app click test (no "Wrong target" warning); ObjectStateChange and re-render proof; Workstream 2 render_and_dump plus precheck on the production viewport; scorecard regression sweep (no scene more than 5 points below Task #69 baseline, well_plate_96_zoom >= 80); naming and style sweep; tsc diff. The matrix in new2_implementation_test_matrix.md lists 16 individual tests across pytest, Playwright, tsc, and eslint.

Open visual-quality blocker. Five non-well-plate scenes lost ground in the Task #97 sweep relative to the Lane C baseline: bench_basic -20, staining_bench -19, drug_dilution_plate_workspace -17, crowded_bench_dense -6, and electrophoresis_bench -47 with 4 hard fails. Resolution work belongs to visual-scorecard repair, not to the runtime-proof workstream.
