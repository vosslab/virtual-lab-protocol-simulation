2. Why NEW0 restarted layout work

The legacy layout engine pre-dated the primary contract item 3 and was the only producer of placement rects. It hard-coded coordinate math (track sizing, gap distribution, content fit) in TypeScript, which made every layout regression a TypeScript-debugger problem rather than a visible CSS problem.

NEW0 retired the in-TS solver in favor of the browser CSS engine. Layout becomes a CSS-cascade question; precheck reads the rendered DOM via getBoundingClientRect(); manifests stay coordinate-free. The trade-off is that NEW0 violates primary contract item 3 by design: the layout engine no longer owns positioning. NEW1 exists to integrate the NEW0 model into production without amending contract item 3, by treating CSS-native layout as the layout engine's output backend (one conditional in adapter.ts, no coordinate math reintroduced).

Lane 3 adapter-only proof. Eleven of eleven assertions pass. The CSS-native adapter actually runs when the override is set; returns a non-empty ComputedItemLayout[] with numeric x, y, width, height; result placement id matches zoom_well_plate_96. Notice this test does NOT load the production bundle: it exercises the spike module in isolation inside a real browser. The gap between this proof and full-runtime proof is what Lanes B-D close.

Lane B built-app render. The scratch harness at dist/_spike_well_plate_96_zoom_check.html loaded the runtime bundle and called SceneRuntime.loadAndMountByProtocolName('well_plate_96_zoom_check'). Lane C, before click. Rendered scene shows scene-viewport with 26 KB of HTML and 122+ elements carrying data-target-id. Notice the spike adapter has populated the layout for the well plate; click dispatch is ready.

8. Remaining blockers

Lane D re-render proof (PARTIAL to IN FLIGHT)

The isTargetSatisfied function at src/scene_runtime/bundle/entry.ts lines 443-491 lives in the forbidden boundary. The current dev_smoke protocol expects well_plate_96.E7, but the validator does not treat E7 as a satisfying member of the named target group.

Path forward: author a new dev_smoke step whose click target is a click-validable parent object (or an object with no sub-parts), so the validator accepts the click and the step completes. This avoids editing the validator and produces a real renderScene re-run for the spike invocation counter to record.

Lane E precheck on built output (BLOCKED to IN FLIGHT)

Precheck at experiments/css_native_layout/precheck.mjs line 1047 calls page.locator('.scene-container').boundingBox() and times out against dist/_spike_well_plate_96_zoom_check.html because the built app emits .scene-container at runtime via the JavaScript bundle, not as static HTML.

Path forward: a new tool, render_and_dump.mjs, that mounts the runtime in a headless browser, waits for .scene-container, snapshots the rendered DOM, and writes a static HTML file. Precheck then ingests the static dump as it already does for the experiments-folder templates. Do not modify precheck itself.

9. Recommended next work

Recommendation: continue Path A through round 2. Two follow-ups, both bounded; neither requires production-architecture change, contract amendment, or coordinate-solver reintroduction.

  1. Lane R (re-render proof). Smallest path: author a dev_smoke step whose click target is a parent object the existing validator accepts. Run spike_built_app_state_change.mjs; confirm the spike invocation counter increments from 1 to 2 after the click.
  2. Lane P (precheck bridge). Write render_and_dump.mjs: mount runtime, wait for .scene-container, snapshot DOM, write static HTML, feed to precheck. Confirm hard-fail count remains 0 against the built-app rendered DOM.

Out of scope for round 2: full migration of the layout module to snake_case identifiers; primary-area-ratio target tuning for electrophoresis; multi-workspace scene switching with the region--instrument_station: display:none rule.
