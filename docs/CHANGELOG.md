## 2026-05-10

### Additions and New Features
- **Patch 1: Browser-first protocol launcher**: `/` now opens a protocol launcher with a primary Cell Culture Protocol card and a separate Tutorials grid. Each launcher entry uses a readable title, optional static description, and a step count computed from generated protocol data. Direct links use `?protocol=<id>` and valid links bypass the launcher.

### Behavior or Interface Changes
- **Protocol generation is catalog-backed**: `tools/build_protocol_data.py` now emits all protocol and inventory data into generated catalogs while preserving the existing `PROTOCOL_ID`, `PROTOCOL_STEPS`, `EQUIPMENT`, and `REAGENTS` facade exports for runtime callers.
- **Walker protocol selection uses direct links**: `tests/playwright/e2e/protocol_walkthrough_yaml.mjs` no longer rebuilds generated protocol data per tutorial. It opens `/?protocol=<id>` for every protocol, including `cell_culture`.

- Added `npm run ui:review`, backed by [tests/playwright/ui_review.mjs](../tests/playwright/ui_review.mjs), to serve the compiled `dist/` build, capture desktop/mobile screenshots, and report browser console/page errors through one approved UI-review command.

### Behavior or Interface Changes
- **YAML is now the single source of truth for bench and hood scene layout**: `src/bench_config.ts` and `src/hood_config.ts` are gone; bench/hood layout is declared in `src/scenes/bench/bench.yaml` and `src/scenes/cell_culture_hood/cell_culture_hood.yaml` and consumed via the layout engine through the `src/scene_configs.ts` facade.
- **`SceneConfig` schema tightened (Patch B12)**: `sceneBounds` and `layoutRules.label*` (label-policy fields) are now formal typed optional fields on `SceneConfig`, promoted out of the prior `[key: string]: unknown` schema-debt. The validator in `tools/build_scene_data.py` rejects unknown top-level keys.
- **Hood and bench dispatch is K2-only**: the legacy compatibility-token ladder (the `<base_tool>_with_<liquid>` shape) folded into K2 `completionPath` dispatch (Patch C1 for hood, Patch C2 for bench). There is one dispatch path per scene and no token-shaped fallback.

### Fixes and Maintenance
- **B10/B11 integrator patch: zones array vs Record collision**: the parallel B10 (bench) and B11 (hood) layout-migration patches landed compatible YAML emitters but incompatible adapter call sites: B10's emitter wrote `zones` as an array while several adapter call sites still treated `zones` as a Record. The integrator patch normalized the conversion at the adapter call sites and repaired the walker regression introduced by the collision; cell_culture walker returned to 25/25.
- **`tests/playwright/build_game_if_missing.mjs` build helper path fix**: the helper was looking at the legacy `cell_culture_game.html` path; updated to `dist-single/game.html` (which is what `export_single_file.sh` actually writes), so Playwright tests no longer fail with a missing-game-file error on a fresh tree.

### Removals and Deprecations
- `src/content/protocol_data.ts`, `src/content/inventory_data.ts`, `src/content/scene_data.ts` removed from `src/`; regenerated builds now write to gitignored `generated/protocol_data.ts`, `generated/inventory_data.ts`, `generated/scene_data.ts` and runtime callers consume them through the `src/protocol.ts`, `src/inventory.ts`, and `src/scene_configs.ts` facades (Plan A).
- `src/bench_config.ts` deleted (bench layout migrated to `src/scenes/bench/bench.yaml`, Patch B10).
- `src/hood_config.ts` deleted (hood layout migrated to `src/scenes/cell_culture_hood/cell_culture_hood.yaml`, Patch B11).
- `src/scenes/shared/legacy_tokens.ts` deleted along with the `buildLegacyToken` API once the last call site moved onto K2 completionPath dispatch (Patch C4).

### Decisions and Failures
- **B10/B11 parallel-dispatch collision**: the Plan B parallel dispatch of B10 (bench layout YAML) and B11 (hood layout YAML) shipped emitters whose YAML output was intra-patch consistent but cross-patch incompatible (zones as array in B10, zones still as Record at multiple adapter call sites that B11 also touched). The walker regression was diagnosed against a clean main-branch checkpoint and resolved by an integrator patch that normalized the call sites. Lesson recorded in the closeout archive: parallel dispatch on a shared schema needs a pre-merge integrator gate, not a per-patch gate.
- **Schema-debt cleared (Patch B12)**: the prior `SceneConfig` carried a `[key: string]: unknown` index signature that allowed `sceneBounds` and `layoutRules.label*` to slip through as untyped escape hatches. B12 promoted both to formal typed optional fields and the index signature was removed.

### Decisions and Failures

- **Lesson from M5 cleanup analysis:** Deletion of "unreachable" legacy code branches must verify completionPath.kind for EACH related step individually, not rely on an outer guard condition. The M5 cleanup initially claimed seven onItemClick branches were "gated by interactionSequence dispatch" and therefore unreachable. This was correct FOR those four specific steps (media_adjust, add_mtt, decant_mtt, add_dmso), which all use `completionPath: kind: interactionSequence`. However, the same reasoning does NOT apply to directTool-dispatch steps (e.g., spray_hood uses `completionPath: kind: directTool` with tool ethanol_bottle). The router only branches to interactionSequence handlers for sequences; directTool and modal steps fall through to the legacy click ladder by design. The ethanol_bottle handler at hood.ts:1028 is load-bearing for tutorial protocols and any future directTool steps, and must not be deleted. Future M-style cleanups must audit the completionPath.kind for each step that references a legacy handler, not just check for an outer sequence guard. This prevents silent regressions where a legacy handler is deleted as "unreachable" even though it's actively used by non-sequence step types.

The fix path was tightening `destination` and adding the direct-tool-interaction concept, NOT adding a fake clickable item to the hood scene.
