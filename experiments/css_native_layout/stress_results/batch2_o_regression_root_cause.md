## Critical finding: Scorecard methodology mismatch

Before per-scene breakdown, a structural issue invalidates the regression framing. Batch1 and batch2_n scorecards were produced by different code paths with different fidelity:

- batch1: Generated by canonical score_layout.mjs running against test-results/new3_stress_batch1/audit/visual_audit.json -- full live DOM measurement with all metrics computed from actual rendered data.
- batch2_n: Generated by /tmp/\_generate_scorecard_batch2_n.py (a temporary helper script) which read precheck_batch2_n/visual_audit.json and HARDCODED four metrics with simplified fixed values:
  - balance: 50 (hardcoded constant, not computed)
  - region_filling: 50 (hardcoded constant, not computed)
  - aspect_ratio_fidelity: 100 (hardcoded constant, not computed)
  - primary_prominence: 100 (hardcoded constant, not computed)

Additionally, precheck_batch2_n/READ_ME_STALE_DATA.md warns that visual_audit.json in that directory contains PRE-fix Batch 1 baseline data, not post-Workstream-N results.

The consequence: every metric that regressed (balance, aspect_ratio_fidelity) was scored differently than in batch1; primary_prominence and region_filling were uniformly inflated to 50/100 instead of actual values.

## Root cause hypothesis (with evidence)

Primary root cause: Scorecard methodology mismatch between batch1 and batch2_n.

Evidence:

1. /tmp/\_generate_scorecard_batch2_n.py lines 112-117: explicit hardcoded constants marked "Simplified" for balance, region_filling, aspect_ratio_fidelity, primary_prominence.
2. precheck_batch2_n/READ_ME_STALE_DATA.md: visual_audit.json contains pre-fix batch1 data, not post-Workstream-N.
3. support_distance in batch1 scorecards = 100 for all 5 scenes. The batch2_n computation reads checks['supporting_distance'].get('mean_norm', 0) but actual precheck field is mean_normalized_distance. The 52.7, 47.2, 49.7, 63.0, 38.3 values come from stale batch1 data already in visual_audit.json.
4. full_comparison.json (authoritative post-fix) shows all 5 scenes at cbp=0, ad_hf=0 -- precheck HARD_FAILs fully resolved by Workstream N.

The apparent 12-16 point regressions are NOT caused by CSS layout changes. They are artifacts of:

- Hardcoded placeholder values inflating primary_prominence, region_filling, aspect_ratio_fidelity
- Stale visual_audit.json input carrying pre-fix supporting_distance values
- Net effect: lower total score even though hard failures (CBP, AD_HF) eliminated

The task brief hypothesis (taller cards reflow composition, push primary out of frame) is NOT supported by the data. primary_area_ratio is identical in all 5 scenes between batch1 and batch2_n. label_overlap changes are minor.

## No scene is marked crowded

None of the 5 YAML files contain scene_density or any crowded attribute. The crowded-density modifier in bench.css (lines 244-284) is not active. Modifier exists in CSS but not being applied.

## Proposed mitigations evaluation

Mitigation (a): Re-engage crowded-density modifier
CSS has [data-scene-density="crowded"] modifier already. Not marked in YAML. Adding requires generator change. Risk: LOW for correctness, MEDIUM for vocabulary correctness. Does not address root cause.

Mitigation (b): Cap rear_shelf max placements per row
No evidence rear_shelf overflow causes the regressions (region_overflow=0 for all 5). Not warranted.

Mitigation (c): Reduce handheld max-height for crowded scenes
The crowded modifier already sets handheld to 112px. Subset of (a). Does not address root cause.

Mitigation (d): Composition-specific .placement rule
No evidence needed. region_overflow=0 for all 5. Root cause is not CSS.

Mitigation (e) NEW: Regenerate batch2_n scorecards using canonical score_layout.mjs
Correct fix. Re-run score_layout.mjs against post-Workstream-N DOM measurement. Apples-to-apples comparison. Requires fresh precheck against rendered scenes with N CSS, then score_layout.mjs. Risk: NONE for layout.

## Recommendation

Do not mitigate via CSS (a, b, c, d). 12-16 point regressions are scorecard methodology artifacts, not CSS layout regressions. CSS changes would fix the symptom, not the design.

Regenerate batch2_n scorecards via canonical score_layout.mjs (mitigation e). Until then, treat "Top 5 Most-Degraded" list in scorecard_batch2_n_HANDOFF.md as unreliable.

Workstream N's win stands. CBP 631->0, AD_HF 570->0, confirmed by full_comparison.json. Footprint max-height changes are correct and should not be reverted.

## Concerns

1. Stale data propagated into a comparison artifact. precheck_batch2_n/visual_audit.json was used for scorecard generation despite stale warning. Future scorecard generation must use fresh precheck output.

2. Scorecard generation used a throwaway helper, not canonical score_layout.mjs. /tmp/\_generate_scorecard_batch2_n.py hardcodes 4 of 9 metrics. Appropriate for quick generation pass but hardcoded values were not flagged as producing incomparable results.

3. gold_drug_dilution_workspace scene class mismatch. YAML: dense_clutter. batch2_n generator: composition (heuristic). Different weight tables compound the methodology mismatch.

4. primary_prominence was 0 in batch1 for all 5 scenes and hardcoded 100 in batch2_n. Inflates batch2_n scores artificially, partially masking the regression. If layout actually improved primary prominence (larger handheld footprints clearer hierarchy), that is a real gain -- cannot be credited without proper measurement.

5. No live render of post-Workstream-N scenes exists for these 5 scenes. precheck_batch2_n PNGs are there but underlying visual_audit.json is stale. True layout quality for these scenes post-N is unmeasured.
