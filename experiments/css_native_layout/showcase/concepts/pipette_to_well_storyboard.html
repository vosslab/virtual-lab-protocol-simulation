CONCEPT DEMO -- NOT PRODUCTION, NOT WIRED TO RUNTIME

Concept: Pipette-to-Well Interaction Storyboard

Four-frame walkthrough of a student transferring 100 uL from a pipette tip into well A1.

Frame 1 / Idle
Pipette held at rack
Student clicks the pipette. Highlight pulses on the tool to acknowledge selection.
Frame 2 / Travel
Cursor-attached pipette
Pipette attaches to the cursor (CursorAttach operation). Motion trail conveys travel; no game-state mutation yet.
Frame 3 / Target
Hover well A1
Validator preset "correct_target" matches; well A1 highlights amber to confirm the right target before commit.
Frame 4 / Commit
Dispense complete
ObjectStateChange writes material_name and material_volume on A1. Visual state renders blue fill. Step resolves; next_step fires.
What this concept shows: a four-beat storyboard for a single pipette-to-well interaction broken into idle, cursor-attached travel, target hover, and commit. Each beat maps to existing primitives in the protocol vocabulary (CursorAttach, ObjectStateChange) and a validator preset (correct_target). The frames make explicit which beats are visual-only and which mutate object state. To realize this in production, the scene runtime would need to animate the cursor-attached pipette with motion trails, render a target-hover highlight that distinguishes "hovered" from "selected," and sequence the dispense visual to follow validator success rather than precede it. None of this requires new vocabulary; it requires wiring existing primitives into the render layer and adding a hover-state visual to the well object's visual_states.
