CONCEPT DEMO -- NOT PRODUCTION, NOT WIRED TO RUNTIME

Concept: Selected Well Highlight States

96-well plate with five distinct well visual states demonstrated simultaneously.

Well visual states

Empty (default)
Filled (material present)
Selected (student picked)
Hovered (cursor over)
Target (next action)
Locked (not interactable)
Error (wrong target)

Where each state lives:

visual_states on the well object declares static renderings (empty, filled).

Runtime-only overlays (selected, hovered, target, error) are scene-side highlights, not declared object state.

"Locked" is an object-level interactive flag toggled by an ObjectStateChange.

What this concept shows: a single 96-well plate displaying all seven candidate well visual states at once, separated into declared object states (empty, filled, locked) and scene-side ephemeral highlights (selected, hovered, target, error). Today the runtime mostly distinguishes only empty vs. filled; adding target and error highlights is the lowest-hanging fruit for student feedback. To realize this in production, the scene runtime would need a small highlight overlay layer keyed to per-target validator outcomes, the well object would need a "locked" interactive flag toggled by ObjectStateChange, and the protocol validator would need to emit a transient "wrong_target" event the highlight layer can subscribe to. None of the highlight states require new vocabulary; they require a render-layer subscription to validator results.
