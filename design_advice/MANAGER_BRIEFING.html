04

When a scene can't render.

Tiers * lint * escape hatch

A scene YAML existing doesn't mean it will render. The convergence loop (Fig C above) catches cluttered - it can't catch broken. Three failure tiers sit beyond the loop's reach. Each needs a different response; the cleanest enforcement point is a build-time design linter, with the runtime engine as the last line of defense.

Design rules the linter would enforce.

Eight rules in v1, each mapping to a structured finding. Errors fail the build; warnings surface in the dev overlay; info-level findings show in author tooling only. Authors can suppress a finding per-scene with a documented reason (lint_suppress: [rule_name]).

"1 / 25 / 2" composition across rows

RECOMMENDATION

Build the linter in Python. Two surfaces, one model.

A Python linter is the right fit for design-quality enforcement: no TS toolchain needed, sub-second feedback while authors edit YAML, and structured findings the runtime can also consume via JSON. The TS validator (M2b item 4) stays focused on hard build gates; the Python linter owns the soft design rules above. Two surfaces, one mental model.

Where it lives
tools/scene_lint.py + tools/scene_lint/rules/ (one file per rule). Reads YAML; emits JSON findings.
When it runs
  • Local: make lint-scenes
  • Pre-commit hook: staged scenes
  • CI: every pull request
What it outputs
JSON to stdout, human-readable to stderr. Exit codes 0 * 1 (warn) * 2 (error). Dev overlay reads the JSON.
Decision needed Should CI gate on errors only, or on errors and warnings? Recommend errors only for v1. Warnings surface in the dev overlay so authors see them, but a row-balance warning shouldn't block a PR. Tighten the policy later if warnings become routine.

The runtime escape-hatch contract.

When the linter is bypassed or skipped - or when the runtime hits Tier 3 anyway - the engine has to respond gracefully. Three rules:

  • Never crash the player. A failed scene must always produce a renderable result of some kind.
  • Walk the extends chain backward. If the protocol-scoped scene fails, render its parent. If the parent fails, render the base scene. If the base fails, render the workspace placeholder.
  • Surface the failure visibly. Warm banner across the top: "Protocol step paused * scene rendering blocked." Telemetry logs the diagnostic that triggered fallback. The player can continue; the protocol step records as skipped_no_render.

02

When does the strict validator ship?

The runtime pipeline does not throw on authoring errors - it accumulates diagnostics. A strict validator (build-time, throws on unknown_object, multi-level extends, locked-field mutations, etc.) is listed as M2b item 4 in the port handoff.

Argument to ship in M2b: catches authoring problems before the renderer crashes mid-development.
Argument to defer to M3: the runtime already surfaces every problem the validator would catch, just later. Authors learn the same lessons either way.

Recommend Defer to M3. The two scenes worth running through M2b will tell us whether the content surface is large enough to warrant a build-time gate. If you'd rather ship it in M2b, it's a 2-day add but it pushes the renderer to week 2.
03

_shrunk_passes debug field: ship in production or dev-only?

Per-placement integer counting how many shrink iterations its zone underwent. The renderer doesn't need it; only debugging and the dev overlay do.

Recommend Keep in the production result. It's one integer per placement and it's invaluable when triaging "why is this item smaller than the cm value says?" on a real scene. Strip it in M3 only if payload size becomes a measured problem.
